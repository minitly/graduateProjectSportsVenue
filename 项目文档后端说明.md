## 体育馆预约与仓库管理系统 - 后端设计说明

本文档面向后端开发与维护人员，说明体育馆预约 + 仓库管理系统在**鉴权、权限模型与相关数据库表**方面的设计方案，重点描述 JWT 无状态认证与拦截器校验流程。

---

## 1. 鉴权与 JWT 方案

- **认证方式**：用户名 + 密码 + 角色登录。
- **Token 类型**：JWT（JSON Web Token）。
- **传输方式**：通过 HTTP 请求头 `Authorization: Bearer <token>` 传递。
- **状态特性**：后端不保存 Session，**纯 JWT，无状态，不保存 Token**。
- **统一拦截**：通过后端拦截器/过滤器，对受保护接口统一进行 Token 校验。

### 1.1 JWT 载荷建议字段

实际实现中，JWT 的 Payload 推荐至少包含以下字段（可根据需要增减）：

- `userId`：用户主键 ID（对应 `sys_user.id`）。
- `username`：用户名。
- `role`：用户角色（USER / OWNER / ADMIN）。
- `exp`：过期时间（Expiration Time）。
- 可选：`tokenVersion`、`loginDevice` 等。

### 1.2 登录与发放 Token 流程

1. 前端调用登录接口 `/sportsVenue/auth/login`，提交 `username`、`password`、`role`。
2. 后端根据 `username` 查询 `sys_user` 表：
   - 若用户不存在，返回 `401 用户名不存在`。
   - 若密码不正确，返回 `401 密码错误`。
   - 若 `role` 与用户实际角色不一致，返回 `403 登录身份不匹配`。
   - 若用户状态为禁用（`status = 0`），返回 `403 账号已被禁用`。
3. 校验通过后，构造 JWT：
   - Header：声明算法（如 HS256）等。
   - Payload：写入 `userId`、`username`、`role`、`exp` 等。
   - Signature：使用服务端配置的密钥进行签名。
4. 将生成的 JWT 作为 `token` 字段返回给前端，同时返回用户基础信息与权限列表。

> 由于不在服务器保存 Token，后端天然支持无状态伸缩部署；多个实例只要共享同一密钥配置即可。

### 1.3 Token 携带与校验流程

1. 前端在访问需要登录的接口时，在请求头中添加：
   - `Authorization: Bearer <JWT_TOKEN>`
2. 后端拦截器/过滤器执行顺序：
   1. 判断当前请求路径是否属于**白名单**（如 `/auth/login`、静态资源等），白名单直接放行。
   2. 从请求头中读取 `Authorization`，解析出 `Bearer ` 后的 Token 字符串。
   3. 校验 Token：
      - 校验签名是否正确。
      - 校验是否过期（`exp`）。
   4. 校验失败：
      - 返回 `401`，提示 Token 无效或已过期。
   5. 校验通过：
      - 将解析得到的 `userId`、`username`、`role` 等信息写入请求上下文（如 ThreadLocal）。
      - 后续业务层根据 `role` 与权限列表做更细粒度的授权控制。

### 1.4 退出登录与强制下线

由于服务端不保存 Token 状态，**普通退出登录逻辑**为：

- 前端删除本地缓存的 Token（localStorage / sessionStorage / Cookie / 内存）。
- 可选：前端删除本地与用户相关的缓存信息。

若需要**服务器端强制下线/失效已发 Token**，可考虑：

- 修改用户状态（如将 `sys_user.status` 置为禁用），在拦截器中解析 Token 后检查用户状态。
- 修改 JWT 签名密钥或引入 `tokenVersion` 字段，版本不匹配则视为失效。
- 配合短有效期 Token + 刷新机制，降低强制失效需求的复杂度。

## 2. 角色与权限模型

权限相关核心表包括：`sys_user`、`sys_permission`、`sys_role_permission`。

### 2.1 角色说明

- **USER**：普通用户
- **OWNER**：场地主/场馆管理者
- **ADMIN**：系统管理员

系统约定：**一个账号只属于一个角色**，不支持多角色混合登录。

### 2.2 权限模型说明

1. 所有可控功能点抽象为权限编码（`permission_code`，如 `USER_BOOK_VENUE`）。
2. 通过 `sys_role_permission` 建立**角色 - 权限**之间的多对多关系。
3. 用户登录后：
   - 根据用户 `role`，从 `sys_role_permission` 中查询对应权限集合。
   - 将权限列表一并返回前端。
   - 前端根据权限控制菜单、按钮的显示与否。

## 3. 数据库表结构概览

以下为 `table.sql` 中与鉴权和权限相关的表结构简要说明。

### 3.1 `sys_user` 表（系统用户表）

```sql
CREATE TABLE sys_user (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户主键ID',
  username VARCHAR(50) NOT NULL UNIQUE COMMENT '用户名',
  password VARCHAR(100) NOT NULL COMMENT '密码',
  real_name VARCHAR(50) NOT NULL COMMENT '真实姓名',
  role VARCHAR(20) NOT NULL COMMENT '角色：USER/OWNER/ADMIN',
  status TINYINT NOT NULL DEFAULT 1 COMMENT '账号状态：1-正常，0-禁用',
  phone VARCHAR(20) DEFAULT NULL COMMENT '手机号',
  email VARCHAR(100) DEFAULT NULL COMMENT '邮箱',
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) COMMENT='系统用户表';
```

字段说明（部分）：

- `username`：登录账号，需唯一。
- `password`：密码（建议实际存储为加盐哈希值）。
- `role`：角色标识（USER / OWNER / ADMIN），与权限控制强绑定。
- `status`：账号状态，拦截器校验 Token 通过后仍需检查该状态。

### 3.2 `sys_permission` 表（系统权限表）

```sql
CREATE TABLE sys_permission (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '权限主键ID',
  permission_code VARCHAR(100) NOT NULL UNIQUE COMMENT '权限编码',
  permission_name VARCHAR(100) NOT NULL COMMENT '权限名称',
  module_name VARCHAR(50) DEFAULT NULL COMMENT '所属模块',
  description VARCHAR(255) DEFAULT NULL COMMENT '权限说明',
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) COMMENT='系统权限表';
```

字段说明（部分）：

- `permission_code`：权限编码，后端与前端共同约定，如 `USER_BOOK_VENUE`。
- `permission_name`：权限中文名称，用于后台配置界面展示。
- `module_name`：可选，标记所属功能模块。

### 3.3 `sys_role_permission` 表（角色权限关联表）

```sql
CREATE TABLE sys_role_permission (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
  role VARCHAR(20) NOT NULL COMMENT '角色：USER/OWNER/ADMIN',
  permission_code VARCHAR(100) NOT NULL COMMENT '权限编码',
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  UNIQUE KEY uk_role_permission (role, permission_code)
) COMMENT='角色权限关联表';
```

设计要点：

- 同一角色与同一权限的组合需唯一（通过联合唯一索引 `uk_role_permission` 保证）。
- 可在初始化脚本中预置不同角色的默认权限集合。

## 4. 与业务模块的后续扩展

当后续增加体育馆预约、仓库管理等业务模块接口时，建议：

1. 在接口层使用注解或统一配置，标记每个接口需要的权限编码（如 `@RequiresPermissions("OWNER_VIEW_VENUE")`）。
2. 拦截器完成基础登录校验后，由权限校验逻辑读取当前用户权限列表，对比接口所需权限，决定是否放行。
3. 在前端根据 `permissions` 字段渲染菜单和按钮，保持前后端权限模型一致。
