## 体育馆预约与仓库管理系统 - 前后端接口文档

本文档面向前后端联调使用，**只保留关键接口信息**，所有鉴权细节、表结构等请参考  `table.sql`。

---

## 一、通用约定

- **服务基础地址**：`http://localhost:9999`
- **应用上下文路径**：`/sportsVenue`
- **完整接口前缀**：`http://localhost:9999/sportsVenue`
- **数据格式**：`application/json`
- **字符编码**：UTF-8
- **统一响应结构**：

```json
{
  "code": 200,
  "message": "OK",
  "data": {}
}
```

- **字段说明**：
  - `code`：业务状态码（200 成功，其他失败）
  - `message`：提示信息
  - `data`：业务数据对象/数组；无数据时可为 `null`

### 1.1 认证与 Token 概述

- **认证方式**：用户名 + 密码 + 角色 登录
- **Token 类型**：JWT
- **传递方式**：请求头 `Authorization: Bearer <token>`
- **特点**：后端不保存 Session，**纯 JWT，无状态**，详情见 `项目文档后端说明.md`

受保护接口（需要登录）必须在请求头中携带 Token：

```text
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.xxx.yyy
```

- **无需鉴权接口**：仅限
  - `POST /sportsVenue/auth/register`
  - `POST /sportsVenue/auth/login`
- **其余所有接口**（例如场地管理、预约管理、仓库管理、用户管理、数据报表、系统管理等模块下的接口）均视为受保护资源，前端必须在请求头中携带当前登录用户的 Token。

## 二、用户注册与登录接口

### 2.1 用户注册接口

#### 2.1.1 接口说明

用于创建系统账号。注册成功后，用户可使用该账号配合角色信息进行登录。

- 普通用户：`USER`（注册**不受限制**）
- 场地主/场地管理者：`OWNER`（注册时必须提供系统 ADMIN 的密码）
- 系统管理员：`ADMIN`（系统唯一账号，由运维/初始化脚本创建，不提供注册入口）

#### 2.1.2 请求信息

- **URL**：`POST /sportsVenue/auth/register`
- **Method**：`POST`
- **Content-Type**：`application/json`

#### 2.1.3 请求参数


| 参数名           | 类型     | 是否必填 | 说明                                                                         |
| ------------- | ------ | ---- | -------------------------------------------------------------------------- |
| username      | String | 是    | 用户名，需唯一                                                                    |
| password      | String | 是    | 登录密码                                                                       |
| realName      | String | 是    | 真实姓名                                                                       |
| role          | String | 是    | 角色：`USER` 或 `OWNER`（不对外开放注册 `ADMIN`）                                       |
| adminPassword | String | 否    | 当 `role = "OWNER"` 时必填，需输入系统唯一 ADMIN 账号的登录密码；当 `role = "USER"` 时前端传空字符串或不传 |
| phone         | String | 否    | 手机号                                                                        |
| email         | String | 否    | 邮箱                                                                         |


#### 2.1.4 请求示例

```json
{
  "username": "zhangsan",
  "password": "123456",
  "realName": "张三",
  "role": "USER",
  "adminPassword": "",
  "phone": "13800000000",
  "email": "zhangsan@example.com"
}
```

OWNER 注册示例（需填写 ADMIN 密码）：

```json
{
  "username": "owner01",
  "password": "123456",
  "realName": "场馆管理员01",
  "role": "OWNER",
  "adminPassword": "Admin真实登录密码",
  "phone": "13800000001",
  "email": "owner01@example.com"
}
```

#### 2.1.5 成功响应示例

```json
{
  "code": 200,
  "message": "注册成功",
  "data": {
    "userId": 1,
    "username": "zhangsan",
    "realName": "张三",
    "role": "USER"
  }
}
```

#### 2.1.6 常见失败响应

- 用户名已存在：

```json
{
  "code": 400,
  "message": "用户名已存在",
  "data": null
}
```

- 角色不允许注册（如尝试注册 ADMIN）：

```json
{
  "code": 400,
  "message": "不支持的注册角色",
  "data": null
}
```

- OWNER 注册时 ADMIN 密码错误：

```json
{
  "code": 403,
  "message": "ADMIN 密码错误，禁止注册场地管理员",
  "data": null
}
```

### 2.2 用户登录接口

#### 2.2.1 接口说明

用于完成系统用户登录认证。前端发送用户名、密码及登录身份，后端校验成功后返回用户信息、访问令牌（JWT Token）及角色拥有的权限列表。前端据此控制页面展示与后续请求。

系统约定：**一个账号只属于一个角色**，不允许同一账号以多个角色身份登录。

#### 2.2.2 请求信息

- **URL**：`POST /sportsVenue/auth/login`
- **Method**：`POST`
- **Content-Type**：`application/json`

#### 2.2.3 请求参数


| 参数名      | 类型     | 是否必填 | 说明                         |
| -------- | ------ | ---- | -------------------------- |
| username | String | 是    | 用户名                        |
| password | String | 是    | 密码                         |
| role     | String | 是    | 登录身份（USER / OWNER / ADMIN） |


#### 2.2.4 请求示例

```json
{
  "username": "zhangsan",
  "password": "123456",
  "role": "USER"
}
```

#### 2.2.5 成功响应示例

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "userId": 1,
    "username": "zhangsan",
    "realName": "张三",
    "role": "USER",
    "token": "eyJhbGciOiJIUzI1NiJ9.xxx.yyy",
    "permissions": [
      "USER_VIEW_VENUE",
      "USER_VIEW_VENUE_DETAIL",
      "USER_BOOK_VENUE"
    ]
  }
}
```

#### 2.2.6 返回字段说明（data）


| 字段名         | 类型     | 说明                      |
| ----------- | ------ | ----------------------- |
| userId      | Long   | 用户主键 ID                 |
| username    | String | 用户名                     |
| realName    | String | 真实姓名                    |
| role        | String | 角色：USER / OWNER / ADMIN |
| token       | String | 登录成功后生成的 JWT Token      |
| permissions | Array  | 当前角色拥有的权限编码列表           |


> `permissions` 为字符串数组，前端据此控制菜单、按钮等展示，具体编码含义可在后端配置或文档中维护。

#### 2.2.7 常见失败响应

- 用户名不存在：

```json
{
  "code": 401,
  "message": "用户名不存在",
  "data": null
}
```

- 密码错误：

```json
{
  "code": 401,
  "message": "密码错误",
  "data": null
}
```

- 登录身份不匹配：

```json
{
  "code": 403,
  "message": "登录身份不匹配",
  "data": null
}
```

- 账号被禁用：

```json
{
  "code": 403,
  "message": "账号已被禁用",
  "data": null
}
```

### 3.0 场地图片上传与获取接口

> 上传与获取均支持**单张和多张**，走同一接口。返回的 url 均用**数组**封装；封面等单张场景时前端取数组第一个元素（如 `urls[0]`、`images[0]`）。

#### 3.0.1 上传接口说明

- 单张或多张图片均使用 `POST /sportsVenue/files/upload`，表单中 `file` 可传多个（同名多 part）。
- 返回 `data.urls` 为路径数组，单张时长度为 1。**封面图等单张场景请前端取 `data.urls[0]` 写入 `coverImageUrl`**；多张时直接将 `data.urls` 写入 `imageUrls`。

#### 3.0.2 上传请求信息

- **URL**：`POST /sportsVenue/files/upload`
- **Method**：`POST`
- **Content-Type**：`multipart/form-data`
- **Header**：`Authorization: Bearer <token>`

#### 3.0.3 上传参数（Form Data）


| 参数名  | 类型     | 是否必填 | 说明                                            |
| ---- | ------ | ---- | --------------------------------------------- |
| file | File   | 是    | 要上传的图片文件；**可传多个**（表单中多个 name="file"），单张即传 1 个 |
| biz  | String | 否    | 业务标识，如 `venue`                                |


#### 3.0.4 上传成功响应示例

```json
{
  "code": 200,
  "message": "上传成功",
  "data": {
    "urls": [
      "/upload/venue/20250318/venue_1742261234567_123.jpg",
      "/upload/venue/20250318/venue_1742261234568_456.jpg"
    ]
  }
}
```

- `data.urls`：**数组**。单张上传时也为长度 1 的数组。**封面等单张设置时，前端取 `data.urls[0]` 写入 `coverImageUrl`**；多张时整数组写入 `imageUrls`。

---

#### 3.0.5 图片获取接口（查单个/列表后拉取图片）

根据上传返回的 path 拉取图片内容，支持**单张或多张**，同一接口。Query 中 `path` 可传多次；返回 `data.images` 数组，单张时长度为 1。

- **URL**：`GET /sportsVenue/files/serve`
- **Method**：`GET`
- **Header**：`Authorization: Bearer <token>`
- **Query 参数**：


| 参数名  | 类型     | 是否必填 | 说明                                                                       |
| ---- | ------ | ---- | ------------------------------------------------------------------------ |
| path | String | 是    | 上传接口返回的 url；**可传多个**（如 `path=/upload/a.jpg&path=/upload/b.jpg`），单张即传 1 个 |


- **成功响应**：JSON，格式与统一响应一致。`data.images` 为数组，每项结构如下：

```json
{
  "code": 200,
  "message": "OK",
  "data": {
    "images": [
      {
        "path": "/upload/venue/20250318/venue_1742261234567_123.jpg",
        "contentType": "image/jpeg",
        "content": "Base64编码后的图片内容字符串..."
      }
    ]
  }
}
```

- 字段说明：
  - `data.images`：数组，单张图片时长度为 1，多张图片时包含多条记录；
  - `path`：与上传接口返回的路径一致；
  - `contentType`：图片的 MIME 类型，如 `image/jpeg`、`image/png` 等；
  - `content`：图片二进制内容的 Base64 字符串。
- **失败响应**：JSON，常见：`400 path 必须以 /upload/ 开头`、`404 文件不存在`。

### 3.1.1 生成场地编号接口

#### 3.1.1.1 接口说明

用于在前端新增场地前，点击“生成编号”按钮时获取一个可用的场地编号字符串。  
后端根据当前时间生成具有业务含义且不重复的编号（实现方式可为时间序列、UUID 或其它规则化编码）。

#### 3.1.1.2 请求信息

- **URL**：`GET /sportsVenue/venues/code/generate`
- **Method**：`GET`
- **Header**：

```text
Authorization: Bearer <登录成功后返回的token>
```

#### 3.1.1.3 请求参数

- 无 Query 参数
- 无请求体（Body）

#### 3.1.1.4 成功响应示例

```json
{
  "code": 200,
  "message": "生成场地编号成功",
  "data": {
    "code": "V20260323193512001"
  }
}
```

字段说明：

- `data.code`：String，生成后的有效场地编号（前端可直接回填到新增表单的 `code` 字段）。

### 3.3 场地信息修改接口

#### 3.3.1 接口说明

根据场地 ID 修改已有场地的基础信息。**前端在编辑时应先查询出完整场地信息，修改后以“完整对象”的形式提交**，即便某些字段未发生变更，也需要原样传回，后端按整个实体进行更新。

后端联动逻辑（实现时必须遵守）：

- 当场地状态从可用（`AVAILABLE`）被修改为不可用状态（`DISABLED`/`MAINTAIN`/`SUSPEND`）时，后端需要**联动取消该场地所有“申请中（APPLIED）”的预约记录**：
  - 将预约状态置为 `CANCELED`
  - 记录取消原因与备注（如：场地停用/维护/暂停预约）
  - 已完成（已核销 VERIFIED）、已取消（CANCELED）、违规（VIOLATION）的记录不处理

#### 3.3.2 请求信息

- **URL**：`PUT /sportsVenue/venues/{id}`
- **Method**：`PUT`
- **Content-Type**：`application/json`
- **Header**：

```text
Authorization: Bearer <登录成功后返回的token>
```

路径参数：


| 参数名 | 说明      |
| --- | ------- |
| id  | 场地主键 ID |


请求体示例（完整实体示例）：

```json
{
  "id": 1,
  "name": "一号篮球场（改名）",
  "code": "V001",
  "type": "篮球场",
  "capacity": 24,
  "price": 150.0,
  "openTime": "08:00",
  "closeTime": "22:00",
  "openTimeDesc": "每日 08:00-22:00",
  "description": "标准室内篮球场，含电子记分牌和观众席。",
  "status": "MAINTAIN",
  "coverImageUrl": "/upload/venue/v001-cover.jpg",
  "imageUrls": [
    "/upload/venue/v001-1.jpg",
    "/upload/venue/v001-2.jpg"
  ],
  "remark": "本周维护，暂停预约。"
}
```

成功响应结构与「场地基本信息录入接口」相同，返回更新后的最新场地信息：

```json
{
  "code": 200,
  "message": "更新场地成功",
  "data": {
    "...": "同上，返回更新后的场地信息"
  }
}
```

#### 3.3.3 常见失败响应

- 场地编号已存在：

```json
{
  "code": 400,
  "message": "场地编号已存在",
  "data": null
}
```

### 3.4 场地信息删除接口

#### 3.4.1 接口说明

根据场地 ID 删除（或逻辑删除）指定场地信息。具体是物理删除还是逻辑删除可在后端实现时确定。

安全校验（实现时必须遵守）：

- 删除场地属于危险操作，前端必须在请求体中传入 `adminPassword`；
- 后端需要校验当前登录用户为 `OWNER/ADMIN` 且 `adminPassword` 与系统 `admin` 账号密码匹配，校验通过后才允许删除；
- 若密码错误/未传，则返回失败。

后端联动逻辑（实现时必须遵守）：

- 删除场地前，后端需要**联动取消该场地所有“申请中（APPLIED）”的预约记录**：
  - 将预约状态置为 `CANCELED`
  - 记录取消原因与备注（如：场地删除）
  - 已完成（已核销 VERIFIED）、已取消（CANCELED）、违规（VIOLATION）的记录不处理

#### 3.4.2 请求信息

- **URL**：`DELETE /sportsVenue/venues/{id}`
- **Method**：`DELETE`
- **Content-Type**：`application/json`
- **Header**：

```text
Authorization: Bearer <登录成功后返回的token>
```

#### 3.4.3 请求参数

路径参数：

对应数据表 `borrow_record`（具体见 `table.sql`），在接口中以 JSON 形式出现的字段如下：


| 字段名               | 类型            | 说明                                                         |
| ----------------- | ------------- | ---------------------------------------------------------- |
| id                | Long          | 借用记录主键 ID                                                  |
| userId            | Long          | 借用人用户 ID，对应 `sys_user.id`                                  |
| itemId            | Long          | 借用的器材 ID，对应 `warehouse_item.id`                            |
| quantity          | Integer       | 借用数量                                                       |
| status            | String        | 借用状态：`REQUESTED`(提出申请)、`USING`(使用中)、`RETURNED`(已归还)        |
| requestedTime     | LocalDateTime | 提出申请时间                                                     |
| approvedTime      | LocalDateTime | 管理员确认借出时间（可为空）                                             |

| depositSnapshot   | Decimal       | 借用时记录的建议押金金额快照（仅用于记录，不做真实支付）                               |
| conditionOnBorrow | String        | 器材状况（借出时，由管理员在确认借出时填写）：`GOOD`(完好)、`DAMAGED`(损坏)、`LOST`(丢失) |
| conditionOnReturn | String        | 器材状况（归还时，由管理员在确认归还时填写）：`GOOD`(完好)、`DAMAGED`(损坏)、`LOST`(丢失) |

| createTime        | LocalDateTime | 记录创建时间                                                     |
| updateTime        | LocalDateTime | 记录更新时间                                                     |


**5.3 分页查询（见 5.3.1 / 5.3.2）专用约定（与上表并存）：**

- **Query 入参**：不使用 `userId`、`itemId`；改用 **`userName`**、**`itemName`**（均为可选字符串，分别对 `sys_user.username`、`warehouse_item.name` 做**模糊匹配**；不传则不对该维度过滤）。
- **分页响应 `records` 中单条记录**：返回 **`userName`**（借用人用户名）、**`itemName`**（器材名称），由后端根据库表 `user_id`、`item_id` 联表查询填充；**不返回** `userId`、`itemId`。
- 其它借用流程接口（如 5.2 申请 / 借出 / 归还）的请求与响应字段仍以**各小节**为准（例如申请时仍可传 `itemId` 等）。

> **器材状况字段约定（conditionOnBorrow / conditionOnReturn）必须使用统一的枚举值**：  
>
> - `GOOD`：完好  
> - `DAMAGED`：损坏  
> - `LOST`：丢失  
> 接口中不得出现“好/优秀”等其它描述，前后端统一按上述枚举值约定。  
> `remark` 字段在不同阶段会被覆盖更新：

## 六、预约管理模块

> 本节定义「场地预约」相关接口。所有接口均为**受保护资源**，必须在 Header 中携带登录成功后的 Token。  
> 预约模块的核心能力包括：查询场地某日/某范围已被占用的时段、用户创建预约、用户取消预约、用户查看自己的预约、OWNER 查看全部预约与核销。

### 6.0 预约规则（后端必须校验）

- **预约粒度**：以 **60 分钟**为单位；`startTime` / `endTime` 必须对齐整点，且 `endTime > startTime`，且时长必须为 60 的倍数。
- **不可跨日**：`startTime` 与 `endTime` 必须在同一自然日内。
- **最多提前 7 天**：今天算第 1 天，可预约日期范围为 [today, today+6]（共 7 天，含今天与第 7 天）。
- **开放时间校验**：预约时间必须落在场地 `openTime` ~ `closeTime` 范围内（以场地表字段为准）。
- **场地状态**：仅当 `venue.status = AVAILABLE` 时允许创建预约；`DISABLED/MAINTAIN/SUSPEND` 均不可预约。
- **并发占用**：同一场地同一时段不可被重复预约；后端需保证并发下强一致（参考占用时段表唯一约束）。
- **取消规则**：
  - 用户仅可取消自己名下、状态为 `APPLIED` 的预约；
  - 若 `cancelTime > startTime - 2小时`，取消仍会导致该条预约被判定为**违规**（状态变更为 `VIOLATION`，并计入本月违规次数）；若 `cancelTime <= startTime - 2小时`，则为正常取消（状态 `CANCELED`）。
- **核销与迟到违规**：
  - 预约开始后 **60 分钟内**用户需到 OWNER 处核销；
  - 若 `now > startTime + 60分钟` 且仍未核销（状态仍为 `APPLIED`），则判定为违规（状态 `VIOLATION`，并计入本月违规次数）；
  - 该类违规由后端定时任务/调度逻辑自动处理（前端无感）。
- **用户预约资格（信用限制）**：
  - 用户本月累计违规次数达到 **3 次**后，取消其本月预约资格；
  - 预约禁用截止时间为下月 1 日 00:00（自然月），到点自动恢复；
  - 当用户处于禁用期（`bookingBannedUntil` 不为空且大于当前时间）时，创建预约直接失败。

---

### 6.1 预约数据结构约定

预约记录在接口中的基础字段如下（对应数据表 `booking_reservation`，具体见 `table.sql`）：


| 字段名           | 类型            | 说明                                             |
| ------------- | ------------- | ---------------------------------------------- |
| id            | Long          | 预约记录主键 ID                                      |
| userId        | Long          | 预约用户 ID                                        |
| venueId       | Long          | 场地 ID                                          |
| startTime     | LocalDateTime | 预约开始时间（整点，60分钟粒度）                              |
| endTime       | LocalDateTime | 预约结束时间（整点，60分钟粒度）                              |

## 七、公告模块

> 本节定义「全站公告」相关接口。所有接口均为**受保护资源**，必须在 Header 中携带登录成功后的 Token。  
> 角色约定：  
>
> - `OWNER`：可创建草稿、修改、逻辑删除、发布/下线、管理端分页查询；  
> - `USER`：仅可查看已发布公告（分页 + 详情）；  
> - 本项目不对 `ADMIN` 提供公告管理能力设计。

### 7.0 公告数据结构约定

公告信息对应数据表 `notice`（具体见 `table.sql`），在接口中以 JSON 形式出现的字段如下：


| 字段名         | 类型            | 说明                                                  |
| ----------- | ------------- | --------------------------------------------------- |
| id          | Long          | 公告主键 ID                                             |
| title       | String        | 公告标题                                                |
| content     | String        | 公告正文（HTML 富文本）                                      |
| status      | String        | 公告状态：`DRAFT`（草稿）/`PUBLISHED`（已发布）/`OFFLINE`（已下线）    |
| publishTime        | LocalDateTime | 发布时间；仅在状态为 `PUBLISHED` 时有值                           |
| isDeleted          | Integer       | 逻辑删除标记：`0` 未删除，`1` 已删除（前端通常无需展示该字段）                 |
| createByUsername   | String        | 创建人**用户名**（`sys_user.username`）；后端由 `create_by` 关联查询填充，**响应中不返回用户 ID** |
| updateByUsername   | String        | 最后更新人**用户名**（`sys_user.username`）；后端由 `update_by` 关联查询填充，**响应中不返回用户 ID** |
| createTime         | LocalDateTime | 创建时间                                                |
| updateTime         | LocalDateTime | 更新时间                                                |


> **说明**：库表仍存 `create_by`、`update_by`（用户 ID）；凡返回公告对象的接口，均将上述 ID 解析为对应账号的 `username` 后以 `createByUsername`、`updateByUsername` 输出。若记录缺失或用户已不存在导致无法关联，对应字段可为 `null`。

公告状态流转约定（后端实现时遵守）：

- 新建公告默认 `DRAFT`；
- `DRAFT/OFFLINE -> PUBLISHED`：允许发布，并写入或更新 `publishTime`；
- `PUBLISHED -> OFFLINE`：允许下线；
- 逻辑删除后（`isDeleted=1`）不再对外展示，也不允许再次发布。

`publishTime` 字段语义约定（后端实现时遵守）：

- `publishTime` 表示“**最新一次发布时间**”；
- 只要公告状态发生“转换到 `PUBLISHED`”，都必须用当前发布时间覆盖 `publishTime`；
- 无论历史发布过多少次，接口只保留最近一次发布对应的 `publishTime`。

## 八、运营与统计模块（V1：前三类）

> 本节先定义三类核心统计接口：总览卡片、预约趋势、场地热度排行。  
> 所有接口均为**受保护资源**，必须在 Header 中携带登录成功后的 Token。  
> 当前角色约定：仅 `OWNER` 可访问。

### 8.0 通用约定

- 统计时间统一按自然日处理（`yyyy-MM-dd`）。
- Query 参数 `startDate`、`endDate` 均必传，且 `startDate <= endDate`。
- 若时间范围内无数据，接口仍返回 `code=200`，统计值为 `0` 或空数组。
- **预约统计口径（8.1 / 8.2 / 8.3 一致）**：凡统计「时间范围内」的预约条数或按天/按场汇总，均以 **`booking_reservation.start_time`（预约开始时间）** 是否落在统计区间内为准：后端将区间处理为 **`[startDate 00:00:00, endDate 次日 00:00:00)`**（左闭右开）。**不按** `create_time`（下单/创建时间）筛选。一条预约记录仅在 **`start_time` 落入该区间** 时计入 `bookingTotal` 及对应状态计数。

---

### 8.1 运营总览统计（Dashboard）

#### 8.1.1 接口说明

用于返回指定时间范围内的核心运营指标，适合前端首页卡片展示。计数规则见 **8.0**（按预约 `start_time` 落入区间，非 `create_time`）。

#### 8.1.2 请求信息

- **URL**：`GET /sportsVenue/reports/dashboard`
- **Method**：`GET`

#### 8.1.3 Query 参数

| --------- | ------ | ---- | -------------------------------- |
| startDate | String | 是    | 统计开始日期，格式 `yyyy-MM-dd`           |
| endDate   | String | 是    | 统计结束日期，格式 `yyyy-MM-dd`（含当日）      |


请求示例：

```text
GET /sportsVenue/reports/dashboard?startDate=2026-03-01&endDate=2026-03-23
```
