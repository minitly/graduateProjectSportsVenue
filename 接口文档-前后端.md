## 体育馆预约与仓库管理系统 - 前后端接口文档

本文档面向前后端联调使用，**只保留关键接口信息**，所有鉴权细节、表结构等请参考 `项目文档后端说明.md` 与 `table.sql`。

---

## 一、通用约定

- **服务基础地址**：`http://localhost:9999`
- **应用上下文路径**：`/sportsVenue`
- **完整接口前缀**：`http://localhost:9999/sportsVenue`
- **数据格式**：`application/json`
- **字符编码**：UTF-8
- **统一响应结构**：

```json
{
  "code": 200,
  "message": "OK",
  "data": {}
}
```

- **字段说明**：
  - `code`：业务状态码（200 成功，其他失败）
  - `message`：提示信息
  - `data`：业务数据对象/数组；无数据时可为 `null`

### 1.1 认证与 Token 概述

- **认证方式**：用户名 + 密码 + 角色 登录
- **Token 类型**：JWT
- **传递方式**：请求头 `Authorization: Bearer <token>`
- **特点**：后端不保存 Session，**纯 JWT，无状态**，详情见 `项目文档后端说明.md`

受保护接口（需要登录）必须在请求头中携带 Token：

```text
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.xxx.yyy
```

- **无需鉴权接口**：仅限
  - `POST /sportsVenue/auth/register`
  - `POST /sportsVenue/auth/login`
- **其余所有接口**（例如场地管理、预约管理、仓库管理、用户管理、数据报表、系统管理等模块下的接口）均视为受保护资源，前端必须在请求头中携带当前登录用户的 Token。

## 二、用户注册与登录接口

### 2.1 用户注册接口

#### 2.1.1 接口说明

用于创建系统账号。注册成功后，用户可使用该账号配合角色信息进行登录。

- 普通用户：`USER`（注册**不受限制**）
- 场地主/场地管理者：`OWNER`（注册时必须提供系统 ADMIN 的密码）
- 系统管理员：`ADMIN`（系统唯一账号，由运维/初始化脚本创建，不提供注册入口）

#### 2.1.2 请求信息

- **URL**：`POST /sportsVenue/auth/register`
- **Method**：`POST`
- **Content-Type**：`application/json`

#### 2.1.3 请求参数

| 参数名         | 类型     | 是否必填 | 说明                                                         |
| ------------ | ------ | ---- |------------------------------------------------------------|
| username     | String | 是    | 用户名，需唯一                                                   |
| password     | String | 是    | 登录密码                                                       |
| realName     | String | 是    | 真实姓名                                                       |
| role         | String | 是    | 角色：`USER` 或 `OWNER`（不对外开放注册 `ADMIN`）                  |
| adminPassword| String | 否    | 当 `role = "OWNER"` 时必填，需输入系统唯一 ADMIN 账号的登录密码；当 `role = "USER"` 时前端传空字符串或不传 |
| phone        | String | 否    | 手机号                                                        |
| email        | String | 否    | 邮箱                                                         |

#### 2.1.4 请求示例

```json
{
  "username": "zhangsan",
  "password": "123456",
  "realName": "张三",
  "role": "USER",
  "adminPassword": "",
  "phone": "13800000000",
  "email": "zhangsan@example.com"
}
```

OWNER 注册示例（需填写 ADMIN 密码）：

```json
{
  "username": "owner01",
  "password": "123456",
  "realName": "场馆管理员01",
  "role": "OWNER",
  "adminPassword": "Admin真实登录密码",
  "phone": "13800000001",
  "email": "owner01@example.com"
}
```

#### 2.1.5 成功响应示例

```json
{
  "code": 200,
  "message": "注册成功",
  "data": {
    "userId": 1,
    "username": "zhangsan",
    "realName": "张三",
    "role": "USER"
  }
}
```

#### 2.1.6 常见失败响应

- 用户名已存在：

```json
{
  "code": 400,
  "message": "用户名已存在",
  "data": null
}
```

- 角色不允许注册（如尝试注册 ADMIN）：

```json
{
  "code": 400,
  "message": "不支持的注册角色",
  "data": null
}
```

- OWNER 注册时 ADMIN 密码错误：

```json
{
  "code": 403,
  "message": "ADMIN 密码错误，禁止注册场地管理员",
  "data": null
}
```

### 2.2 用户登录接口

#### 2.2.1 接口说明

用于完成系统用户登录认证。前端发送用户名、密码及登录身份，后端校验成功后返回用户信息、访问令牌（JWT Token）及角色拥有的权限列表。前端据此控制页面展示与后续请求。

系统约定：**一个账号只属于一个角色**，不允许同一账号以多个角色身份登录。

#### 2.2.2 请求信息

- **URL**：`POST /sportsVenue/auth/login`
- **Method**：`POST`
- **Content-Type**：`application/json`

#### 2.2.3 请求参数

| 参数名      | 类型     | 是否必填 | 说明   |
| -------- | ------ | ---- | ---- |
| username | String | 是    | 用户名  |
| password | String | 是    | 密码   |
| role     | String | 是    | 登录身份（USER / OWNER / ADMIN） |

#### 2.2.4 请求示例

```json
{
  "username": "zhangsan",
  "password": "123456",
  "role": "USER"
}
```

#### 2.2.5 成功响应示例

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "userId": 1,
    "username": "zhangsan",
    "realName": "张三",
    "role": "USER",
    "token": "eyJhbGciOiJIUzI1NiJ9.xxx.yyy",
    "permissions": [
      "USER_VIEW_VENUE",
      "USER_VIEW_VENUE_DETAIL",
      "USER_BOOK_VENUE"
    ]
  }
}
```

#### 2.2.6 返回字段说明（data）

| 字段名        | 类型            | 说明                 |
| ----------- | ------------- | ------------------ |
| userId      | Long          | 用户主键 ID           |
| username    | String        | 用户名                |
| realName    | String        | 真实姓名               |
| role        | String        | 角色：USER / OWNER / ADMIN |
| token       | String        | 登录成功后生成的 JWT Token |
| permissions | Array<String> | 当前角色拥有的权限编码列表     |

> `permissions` 为字符串数组，前端据此控制菜单、按钮等展示，具体编码含义可在后端配置或文档中维护。

#### 2.2.7 常见失败响应

- 用户名不存在：

```json
{
  "code": 401,
  "message": "用户名不存在",
  "data": null
}
```

- 密码错误：

```json
{
  "code": 401,
  "message": "密码错误",
  "data": null
}
```

- 登录身份不匹配：

```json
{
  "code": 403,
  "message": "登录身份不匹配",
  "data": null
}
```

- 账号被禁用：

```json
{
  "code": 403,
  "message": "账号已被禁用",
  "data": null
}
```

## 三、场地管理模块

### 3.0 场地图片上传与获取接口

> 上传与获取均支持**单张和多张**，走同一接口。返回的 url 均用**数组**封装；封面等单张场景时前端取数组第一个元素（如 `urls[0]`、`images[0]`）。

#### 3.0.1 上传接口说明

- 单张或多张图片均使用 `POST /sportsVenue/files/upload`，表单中 `file` 可传多个（同名多 part）。
- 返回 `data.urls` 为路径数组，单张时长度为 1。**封面图等单张场景请前端取 `data.urls[0]` 写入 `coverImageUrl`**；多张时直接将 `data.urls` 写入 `imageUrls`。

#### 3.0.2 上传请求信息

- **URL**：`POST /sportsVenue/files/upload`
- **Method**：`POST`
- **Content-Type**：`multipart/form-data`
- **Header**：`Authorization: Bearer <token>`

#### 3.0.3 上传参数（Form Data）

| 参数名 | 类型   | 是否必填 | 说明                                                         |
|--------|--------|----------|--------------------------------------------------------------|
| file   | File   | 是       | 要上传的图片文件；**可传多个**（表单中多个 name="file"），单张即传 1 个 |
| biz    | String | 否       | 业务标识，如 `venue`                                         |

#### 3.0.4 上传成功响应示例

```json
{
  "code": 200,
  "message": "上传成功",
  "data": {
    "urls": [
      "/upload/venue/20250318/venue_1742261234567_123.jpg",
      "/upload/venue/20250318/venue_1742261234568_456.jpg"
    ]
  }
}
```

- `data.urls`：**数组**。单张上传时也为长度 1 的数组。**封面等单张设置时，前端取 `data.urls[0]` 写入 `coverImageUrl`**；多张时整数组写入 `imageUrls`。

---

#### 3.0.5 图片获取接口（查单个/列表后拉取图片）

根据上传返回的 path 拉取图片内容，支持**单张或多张**，同一接口。Query 中 `path` 可传多次；返回 `data.images` 数组，单张时长度为 1。**封面等单张时前端取 `images[0]`，用其 `content`（Base64）拼接为 data URL 展示。**

- **URL**：`GET /sportsVenue/files/serve`
- **Method**：`GET`
- **Header**：`Authorization: Bearer <token>`

- **Query 参数**：

| 参数名 | 类型   | 是否必填 | 说明                                                                 |
|--------|--------|----------|----------------------------------------------------------------------|
| path   | String | 是       | 上传接口返回的 url；**可传多个**（如 `path=/upload/a.jpg&path=/upload/b.jpg`），单张即传 1 个 |

- **成功响应**：JSON，格式与统一响应一致。`data.images` 为数组，每项为 `{ path, contentType, content }`，其中 `content` 为 Base64 编码的图片内容。单张时 `images` 长度为 1，**封面等单张场景请取 `data.images[0]`**。前端展示示例：

```text
data:${contentType};base64,${content}
```

即 `"data:" + data.images[0].contentType + ";base64," + data.images[0].content` 作为 `<img src="...">` 即可。

- **失败响应**：JSON，常见：`400 path 必须以 /upload/ 开头`、`404 文件不存在`。

### 3.1 场地信息管理 - 数据结构约定

场地信息在接口中的基础字段如下（对应数据表 `venue`，具体见 `table.sql`）：

| 字段名            | 类型      | 说明                                               |
|-----------------|---------|--------------------------------------------------|
| id              | Long    | 场地主键 ID                                        |
| name            | String  | 场地名称                                            |
| code            | String  | 场地编号（系统内唯一标识）                                  |
| type            | String  | 场地类型（如：篮球场、羽毛球馆、游泳馆等，自由字符串或后续做字典）           |
| capacity        | Integer | 容纳人数                                            |
| price           | Decimal | 收费标准（单位元/小时，接口中用 Number 表示）                    |
| openTimeDesc    | String  | 开放时间说明（如：每天 8:00-22:00，节假日特殊说明等）                 |
| description     | String  | 场地描述                                            |
| status          | String  | 场地状态：`AVAILABLE`(空闲可用)、`DISABLED`(已停用)、`MAINTAIN`(维护中)、`SUSPEND`(暂停预约) |
| coverImageUrl   | String  | 封面图片 URL，通常为上传后后端返回的相对或绝对路径                    |
| imageUrls       | Array   | 图片 URL 集合，前端上传多张图片后由后端生成路径并返回               |
| remark          | String  | 备注信息                                            |

### 3.2 场地基本信息录入接口

#### 3.2.1 接口说明

用于录入新的场地基本信息，包括名称、编号、类型、容量、收费标准、开放时间说明、状态、图片等。录入成功后返回该场地的完整信息。

#### 3.2.2 请求信息

- **URL**：`POST /sportsVenue/venues`
- **Method**：`POST`
- **Content-Type**：`application/json`
- **Header**：

```text
Authorization: Bearer <登录成功后返回的token>
```

#### 3.2.3 请求参数（Body）

```json
{
  "name": "一号篮球场",
  "code": "V001",
  "type": "篮球场",
  "capacity": 20,
  "price": 120.0,

  "coverImageUrl": "/upload/venue/v001-cover.jpg",
  "imageUrls": [
    "/upload/venue/v001-1.jpg",
    "/upload/venue/v001-2.jpg"
  ],
  "remark": "优先开放给校队训练，周末对外开放。"
}
```

字段说明：

| 字段名         | 是否必填 | 说明                                    |
|--------------|------|---------------------------------------|
| name         | 是    | 场地名称                                 |
| code         | 是    | 场地编号，需在系统内唯一                         |
| type         | 是    | 场地类型                                 |
| capacity     | 否    | 容纳人数                                 |

| openTimeDesc | 否    | 开放时间说明                               |
| description  | 否    | 场地描述                                 |
| status       | 是    | 场地状态：`AVAILABLE`/`DISABLED`/`MAINTAIN`/`SUSPEND` |
| coverImageUrl| 否    | 封面图片 URL（前端先上传图片，后端返回路径后再写入）      |
| imageUrls    | 否    | 图片 URL 列表（同上）                        |
| remark       | 否    | 备注信息                                 |

#### 3.2.4 成功响应示例

### 3.3 场地信息修改接口

#### 3.3.1 接口说明

根据场地 ID 修改已有场地的基础信息。**前端在编辑时应先查询出完整场地信息，修改后以“完整对象”的形式提交**，即便某些字段未发生变更，也需要原样传回，后端按整个实体进行更新。

#### 3.3.2 请求信息

- **URL**：`PUT /sportsVenue/venues/{id}`
- **Method**：`PUT`
- **Content-Type**：`application/json`
- **Header**：

```text
Authorization: Bearer <登录成功后返回的token>
```

路径参数：

| 参数名 | 说明     |
|-----|--------|
| id  | 场地主键 ID |

请求体示例（完整实体示例）：

```json
{
  "id": 1,
  "name": "一号篮球场（改名）",
  "code": "V001",
  "type": "篮球场",
  "capacity": 24,
  "price": 150.0,
  "openTimeDesc": "每日 08:00-22:00",
  "description": "标准室内篮球场，含电子记分牌和观众席。",
  "status": "MAINTAIN",
  "coverImageUrl": "/upload/venue/v001-cover.jpg",
  "imageUrls": [
    "/upload/venue/v001-1.jpg",
    "/upload/venue/v001-2.jpg"
  ],
  "remark": "本周维护，暂停预约。"
}
```

成功响应结构与「场地基本信息录入接口」相同，返回更新后的最新场地信息：

```json
{
  "code": 200,
  "message": "更新场地成功",
  "data": {
    "...": "同上，返回更新后的场地信息"
  }
}
```
